# 유닉스 프로세스·스레드·메모리 분석 보고서

> 대상: 개인 프로젝트 서버 **공주비서(Princess Secretary)** — 디스코드 집사봇
> 작성: 2026-07-22 / 환경: macOS (Darwin 25.5.0, arm64), Python 3.11.15, Node v26
> 과제 트랙 Phase 2 (`PLAN.md` §9)

## 1. 분석 대상은 무엇인가

공주비서는 디스코드에 상주하며, 사용자가 자연어로 말하면 노션을 대신 채워주는 봇이다.
껐다 켜는 프로그램이 아니라 **계속 살아있는 프로세스(daemon)**라서 프로세스 분석 대상으로 적합하다.

```
run.py
  └─ secretary/bot.py  main()          ← 디스코드 게이트웨이(WSS) 연결, 메시지 대기
       ├─ AsyncSqliteSaver             ← data/memory.sqlite 에 대화기억 저장
       └─ secretary/agent.py           ← Claude + 도구 묶음(LangGraph)
            ├─ tools.py                ← npx로 노션 MCP 서버를 자식 프로세스로 띄움
            └─ notion_tools.py         ← httpx로 노션 REST 직접 호출
```

이 봇이 교과서적 분석 대상인 이유 세 가지:

1. **상주하며 대기한다** → 유휴 상태와 작업 상태를 나눠 비교할 수 있다
2. **asyncio 기반이라 "스레드가 몇 개인가"가 반직관적이다** → 프로세스 vs 스레드 설명에 적합
3. **작업할 때마다 자식 프로세스를 낳고 곧바로 죽인다** → 프로세스 생성·소멸을 실물로 관찰 가능

### 측정 방법

`scripts/inspect_process.sh` 를 만들어 세 시점에 **동일한 명령 묶음**을 실행했다.
손으로 명령을 치면 (1) 시점마다 옵션이 미묘하게 달라져 비교가 무의미해지고
(2) 뒤에서 보듯 자식 프로세스가 1~2초 만에 사라져 포착을 놓치기 때문이다.

| 스냅샷 | 시점 | 원본 |
|---|---|---|
| A | 유휴 (봇 켜고 대기만) | `reports/raw/A-idle.txt` |
| B | 작업 중 (노션 도구 호출) | `reports/raw/B-working.txt` |
| C | 작업 후 | `reports/raw/C-after.txt` |
| — | 자식 프로세스 생성·소멸 타임라인 | `reports/raw/B-children-timeline.txt` |

> ⚠️ **측정 조건 고지.** A는 1차 실행(PID 75472), B·C는 2차 실행(PID 96826)에서 채집했다.
> 실습 도중 봇에서 별개의 버그가 발견되어(§8) 코드를 고치고 재기동했기 때문이다.
> 같은 코드·같은 기계지만 프로세스가 다르므로, A↔B/C의 절대 수치 비교는
> 참고값으로만 읽어야 한다. 이 한계는 §6에서 실제로 문제가 된다.

## 2. 실습 환경과 macOS 명령어 대응표

교재와 대부분의 자료는 리눅스 기준이다. 그런데 **macOS에는 `/proc` 파일시스템이 없다.**
리눅스가 "프로세스 정보를 가짜 파일로 보여주는" 방식을 택한 반면, macOS(XNU 커널)는
`sysctl`·Mach 태스크 포트를 통해 정보를 제공한다. 따라서 명령을 갈아끼워야 한다.

| 목적 | 리눅스 | macOS (이번에 쓴 것) |
|---|---|---|
| 프로세스 목록·상태 | `ps -ef`, `pstree` | `ps -ef`, `ps -o ...` |
| 스레드 목록 | `top -H`, `/proc/<pid>/task` | **`ps -M <pid>`** |
| 가상 메모리 지도 | `pmap`, `/proc/<pid>/maps` | **`vmmap <pid>`** |
| 메모리 요약 | `/proc/<pid>/status` | `ps -o rss,vsz`, `footprint` |
| 열린 파일·소켓 | `lsof`, `ss`, `netstat` | `lsof` (동일), `nettop` |
| 실행 추적 | `strace` | `sample`, `dtruss` (SIP 제한) |

실습 중 만난 사소한 차이 하나: `date '+%H:%M:%S.%1N'`의 밀리초 포맷 `%N`은 GNU 확장이라
macOS의 BSD `date`에서는 그냥 문자 `N`으로 출력된다. 감시 스크립트를 초 단위로 낮춰야 했다.

> Phase 5에서 EC2(리눅스)에 배포한 뒤 같은 분석을 재실행해 비교 절을 추가할 예정이다.

### 보안 주의 — 보고서에 절대 넣으면 안 되는 명령

`ps -E` 는 프로세스의 **환경변수를 전부 출력**한다. 이 봇은 `DISCORD_BOT_TOKEN`,
`ANTHROPIC_API_KEY`, `NOTION_TOKEN` 을 환경변수로 받으므로, `ps -E` 결과를 보고서에 붙이면
**토큰이 공개 저장소에 그대로 올라간다.** 리눅스의 `/proc/<pid>/environ` 도 마찬가지다.
`reports/` 는 `.gitignore` 대상이 아니므로 이 스크립트에서는 `-E`를 의도적으로 배제했다.

## 3. 프로세스

### 3.1 기본 정보 (스냅샷 C)

```
$ ps -o pid,ppid,rss,vsz,%cpu,%mem,stat,etime,command -p 96826
  PID  PPID    RSS      VSZ  %CPU %MEM STAT ELAPSED COMMAND
96826 96824 145360 435508736   0.0  0.8 S      07:29 .venv/bin/python -u run.py
```

- **PID 96826 / PPID 96824** — 부모는 봇을 실행한 쉘이다. 프로세스는 반드시 부모가 있고,
  부모가 `fork()`로 자신을 복제한 뒤 `exec()`로 파이썬으로 갈아입는 방식으로 태어난다.
- **STAT = S (sleeping)** — 세 스냅샷 모두 `S`였다. `%CPU`도 0.0~0.1.
  **일을 안 해서가 아니라, 대부분의 시간을 "메시지가 오기를 기다리며" 커널에 잠들어 있기 때문**이다.
  CPU를 태우는 계산 작업(CPU bound)이 아니라 네트워크 응답을 기다리는 **I/O bound** 프로세스다.
- **ELAPSED** — 켜진 뒤 흐른 시간. 상주 프로세스임을 보여준다.

### 3.2 자식 프로세스 — 생성과 소멸의 실측

이 봇의 가장 흥미로운 지점이다. 노션 도구를 호출할 때마다 `npx`로 노션 MCP 서버가
**새로 뜬다.** 그런데 세 스냅샷 모두에서 자식 프로세스는 **0개**로 나왔다.

```
--- [3] 자식 프로세스 (PPID = 96826 인 것만) ---
→ 직계 자식 없음
```

수명이 너무 짧아 손으로 명령을 치는 사이에 이미 죽었기 때문이다. 그래서 **0.2초 간격으로
프로세스 목록을 훑는 감시 스크립트**(`scripts/watch_children.sh`)를 따로 만들어 포착했다.

```
시각        이벤트   PID   부모    명령
22:21:48  🟢 생성   3265  96826  npm
22:21:49  🟢 생성   3307   3265  node ... notion-mcp-server
22:21:50  🔴 소멸   3265, 3307
22:21:54  🟢 생성   3541  96826  npm exec @notionhq/notion-mcp-server
22:21:55  🟢 생성   3574   3541  node ... notion-mcp-server
22:21:55  🔴 소멸   3541, 3574
22:21:59  🟢 생성   3737 / 3765          (3번째)
22:22:00  🔴 소멸
22:22:04  🟢 생성   3955 / 3982          (4번째)
22:22:05  🔴 소멸
```

**17초 동안 프로세스 8개가 태어나고 죽었다.** 읽어낼 수 있는 것:

1. **2단 프로세스 트리** — `봇(96826) → npm exec(3265) → node notion-mcp-server(3307)`.
   봇의 직계 자식은 `npm`이라는 **런처**이고, 실제 일을 하는 노션 서버는 **손자**다.
   그래서 자식만 세면 실체를 놓친다. 감시 스크립트가 손자까지 추적하도록 만든 이유다.
2. **호출 1번 = 프로세스 1쌍** — 도구를 4번 호출했고 정확히 4쌍이 생성·소멸했다.
   MCP 세션이 호출마다 새로 열린다는 설계가 실측으로 확인됐다.
3. **수명 1~2초** — 태어나서 stdio로 요청 하나 처리하고 즉시 종료한다.
4. **약 5초의 공백** — 한 쌍이 죽고 다음이 뜨기까지의 간격. 이 시간에 Claude가 도구 결과를
   읽고 다음 행동을 판단한다. 즉 **공백 = LLM 추론 시간, 생성 구간 = 도구 실행 시간**으로
   런타임 파이프라인의 리듬이 프로세스 목록에 그대로 찍힌다.
5. **좀비·고아 없음** — 모든 자식이 부모에게 정상 회수됐다(`Z` 상태 없음).
   봇 종료 후 확인에서도 남은 자식은 없었다.

**성능적 함의**: 도구 호출 1번마다 Node 런타임을 새로 띄우는 비용(콜드 스타트)이 든다.
관측된 1~2초 중 상당 부분이 이 기동 시간이다. CLAUDE.md에도 "첫 노션 호출이 약간 느림 —
뼈대 단계라 의도적"으로 기록돼 있으며, 이 보고서는 그 비용이 **호출마다 반복적으로**
발생함을 수치로 확인했다. MCP 세션을 상주시키면 제거할 수 있는 병목이다.

### 3.3 프로세스 종료와 신호 (실습 중 관찰)

봇을 끄면서 종료코드 **143**을 관찰했다. 유닉스는 신호로 종료된 프로세스의 종료코드를
**128 + 신호번호**로 보고한다.

| 종료코드 | 의미 |
|---|---|
| 0 | 스스로 정상 종료 |
| 130 = 128+2 | SIGINT — 터미널에서 Ctrl+C |
| **143 = 128+15** | **SIGTERM — "얌전히 끝내라". 이번에 관찰한 값** |
| 137 = 128+9 | SIGKILL — 강제 종료. 프로세스가 뒷정리할 기회가 없다 |

SIGTERM은 프로그램이 가로채서 뒷정리(파일 닫기, 연결 종료)를 할 수 있지만 SIGKILL은 불가능하다.
이 봇은 SQLite에 계속 기록하므로 `kill -9`는 대화기억 파일 손상 위험이 있어 마지막 수단이어야 한다.

또 하나 관찰: 봇 프로세스가 죽은 뒤에도 **디스코드에는 1~3분간 온라인으로 표시**됐다.
게이트웨이가 heartbeat 누락을 몇 번 확인한 뒤에야 연결을 끊기 때문으로, 프로세스의 죽음과
원격 서버가 인지하는 시점 사이에 시차가 있다.

## 4. 스레드

```
$ ps -M 96826
USER         PID   TT   %CPU STAT PRI     STIME     UTIME COMMAND
hwangsubin 96826   ??    0.0 S    31T   0:00.28   0:01.25 .venv/bin/python -u run.py
           96826         0.0 S    31T   0:00.01   0:00.00
           96826         0.0 S    31T   0:00.00   0:00.00
           ... (총 8줄)
```

| 스냅샷 | 스레드 수 |
|---|---|
| A (유휴) | **4** |
| B (작업 중) | **8** |
| C (작업 후) | **8** |

### 4.1 asyncio인데 왜 스레드가 여럿인가

이 봇은 `asyncio` 기반이라 **동시성을 스레드로 구현하지 않는다.** 이벤트 루프 하나가
"디스코드 응답 대기" "노션 응답 대기"를 번갈아 처리한다. 그렇다면 스레드는 1개여야 할 것 같은데
실제로는 4~8개다. 이유는:

- **파이썬 GIL** 때문에 파이썬 바이트코드는 어차피 한 번에 한 스레드만 실행한다.
  그래서 추가 스레드는 계산을 나눠 하려는 게 아니라 **블로킹 작업을 이벤트 루프 밖으로
  내보내기 위한 것**이다.
- 대표적으로 **SQLite 접근**. `AsyncSqliteSaver`는 비동기 인터페이스를 제공하지만 실제 디스크
  I/O는 블로킹이므로 별도 스레드 풀에서 처리한다. 대화기억을 저장하는 순간이 여기 해당한다.
- 그 외 DNS 조회, TLS 핸드셰이크 등도 스레드 풀로 넘어간다.

작업 시 4 → 8로 늘어난 것이 이를 뒷받침한다. **일이 들어오자 파이썬이 스레드 풀을 확장했고,
작업이 끝난 뒤(C)에도 8개가 유지됐다** — 스레드 풀은 다음 요청을 위해 만든 스레드를 재사용하려고
바로 반납하지 않는다. 프로세스와 달리 스레드는 **생성 비용이 싸고 회수도 게으르다.**

### 4.2 프로세스와 스레드의 차이가 이 봇에서 드러나는 방식

같은 봇 안에서 두 가지가 동시에 관찰됐다.

| | 자식 프로세스 (노션 MCP) | 스레드 (SQLite 등) |
|---|---|---|
| 메모리 | **분리됨.** 별도 주소 공간 | **공유.** 같은 힙을 함께 씀 |
| 통신 방법 | stdio 파이프 (직렬화 필요) | 변수 직접 접근 |
| 생성 비용 | 비쌈 (Node 런타임 기동 1~2초) | 쌈 |
| 수명 | 1~2초 쓰고 버림 | 만들어두고 재사용 |
| 죽으면 | 봇은 멀쩡함 | 봇 전체가 위험 |

노션 서버를 **프로세스로** 띄운 것은 남이 만든 Node 프로그램이라 어쩔 수 없기도 하지만,
격리 측면에서 이점이 있다. 그것이 죽거나 폭주해도 봇 프로세스는 영향을 받지 않는다.
비용은 매번 치르는 1~2초의 기동 시간이다. **"격리를 사고 속도를 지불한" 교환**이다.

## 5. 메모리

### 5.1 VSZ vs RSS — 415GB의 정체

```
RSS = 145,360 KB  (약 142 MB)
VSZ = 435,508,736 KB  (약 415 GB)
```

VSZ가 RSS의 **약 3,000배**다. 이 기계에는 415GB 메모리가 없다. VSZ는 "실제로 쓰는 양"이 아니라
**"주소 공간에 예약해 둔 범위"**이기 때문이다. 64비트 주소 공간은 사실상 무한하므로, 라이브러리와
런타임이 "일단 크게 잡아두고 필요할 때 쓰는" 전략을 취한다. 잡아만 두고 건드리지 않은 페이지에는
물리 메모리가 배정되지 않는다.

결론: **macOS에서 VSZ는 메모리 사용량 지표로 쓸 수 없다.** 의미 있는 값은 RSS다.

### 5.2 vmmap — 메모리를 무슨 용도로 쓰는가 (스냅샷 C)

```
영역                  VIRTUAL   RESIDENT    DIRTY
__TEXT                311.3M     139.4M       0K     실행 코드 (파이썬 + 확장모듈)
__LINKEDIT            576.7M      10.6M       0K     링커 정보 (거의 안 올라옴)
__OBJC_RO              79.1M      40.7M       0K     macOS 시스템 프레임워크
MALLOC_SMALL           68.0M      47.4M      43.2M   파이썬 객체들이 사는 힙
VM_ALLOCATE            65.1M      65.1M      65.1M
Stack                 104.7M       272K      272K    스레드 스택 (8개 영역)
------------------------------------------------
TOTAL                   1.4G     328.3M     113.7M
```

읽어낼 점 세 가지:

1. **`Stack` 영역이 정확히 8개** — §4에서 센 스레드 8개와 일치한다.
   **스레드마다 자기 스택을 갖는다**는 사실이 메모리 지도에 그대로 찍혔다.
   그런데 104.7MB를 예약해두고 실제로 쓰는 건 **272KB뿐**이다(0.25%).
   VSZ가 왜 그렇게 큰지 보여주는 축소판이다.
2. **DIRTY 113.7MB가 진짜 비용** — `__TEXT`나 `__LINKEDIT`는 디스크의 파일을 그대로 올린
   것이라 메모리가 부족하면 그냥 버렸다가 다시 읽으면 된다(dirty 0K). 반면 힙(`MALLOC_SMALL`)은
   프로그램이 값을 써넣었으므로 버릴 수 없다. **"실제로 이 프로세스가 붙잡고 있는 메모리"**는
   RESIDENT가 아니라 DIRTY에 가깝다.
3. **힙 상세** — `DefaultMallocZone`에 **할당 111,491건, 41.0MB**, 단편화 11%.
   파이썬 객체가 11만 개 살아있다는 뜻이다. LangGraph가 대화 상태를 통째로 들고 있는 구조를 생각하면
   자연스러운 규모다.

### 5.3 ps의 RSS(142MB)와 vmmap의 RESIDENT(328MB)가 다른 이유

같은 시점인데 값이 2배 이상 차이 난다. vmmap의 RESIDENT는 **다른 프로세스와 공유하는 페이지**
(시스템 프레임워크, 공용 라이브러리)까지 포함해 세는 반면, `ps`의 RSS는 다르게 집계하기 때문이다.
`__OBJC_RO` 40.7MB 같은 시스템 프레임워크는 기계 안 모든 앱이 같은 물리 페이지를 함께 본다.
**"이 프로세스 때문에 늘어난 메모리"를 알고 싶다면 공유분을 빼야 한다.**
메모리 수치는 어떤 도구로 쟀는지를 함께 밝히지 않으면 의미가 없다.

### 5.4 RSS 추이 — 예상과 어긋난 관측

| | A (유휴) | B (작업 중) | C (작업 후) |
|---|---|---|---|
| **2차 실행** (PID 96826) | — | 138 MB | 142 MB |
| **1차 실행** (PID 75472) | 127 MB | 61 MB | 31 MB (종료 직전) |

1차 실행에서 메모리가 **줄어들었다.** "일하면 늘어난다"는 직관과 반대다.
정직하게 적자면 **이 두 추이는 서로 모순되며, 원인을 확정하지 못했다.** 다만 조건이 달랐다:

- 1차 실행의 모든 요청은 §8의 버그로 **API 호출 직전에 실패**했다. 즉 노션 응답을 받아 파싱하는
  실제 작업이 없었고, 봇은 대부분 유휴 상태였다.
- macOS는 메모리 압박이 있을 때 유휴 프로세스의 페이지를 회수하거나 압축한다.
  기동 직후 잔뜩 올라왔던 페이지가 시간이 지나며 회수된 것으로 보인다.
- 2차 실행은 노션 응답(한 번에 15KB가 넘는 JSON 포함)을 실제로 파싱했고, 그만큼 힙이 늘었다.

**교훈**: RSS는 프로그램이 "요구한" 양이 아니라 OS가 **"지금 물리 메모리에 얹어 둔" 양**이라
프로그램 바깥 사정(전체 메모리 압박)에 흔들린다. 한 번의 측정으로 추세를 단정하면 안 된다.
동일 조건에서 반복 측정이 필요하며, 이는 §1에서 고지한 측정 한계(A와 B/C가 다른 실행)와도 겹친다.

## 6. 파일 디스크립터와 소켓

### 6.1 열린 파일 — SQLite의 흔적

```
FD  6u  /Users/hwangsubin/project/data/memory.sqlite       (729 KB)
FD 12u  /Users/hwangsubin/project/data/memory.sqlite-wal   (2.6 MB)
FD 14u  /Users/hwangsubin/project/data/memory.sqlite-shm   (32 KB)
```

파일 하나가 아니라 **3개**가 열려 있다. SQLite의 **WAL(Write-Ahead Logging)** 모드 때문이다.
본체를 직접 고치는 대신 변경분을 `-wal`에 먼저 append하고, `-shm`은 여러 접속이 그 위치를
공유하는 데 쓴다. 덕분에 읽기와 쓰기가 서로를 막지 않는다.

주목할 점은 **`-wal`(2.6MB)이 본체(729KB)보다 크다는 것.** LangGraph는 매 스텝마다 대화 상태를
**통째로** 스냅샷 저장한다. 메시지 20개짜리 대화면 20개짜리 스냅샷을, 21개가 되면 21개짜리를
또 통째로 쓴다. 그래서 **대화 길이의 제곱에 비례해** 파일이 커진다.
실제로 이전 DB(`data/memory1.sqlite`)는 체크포인트 205장에 **49MB**였고,
가장 큰 스냅샷 하나가 453KB였다.

단, 이는 **디스크 비용일 뿐 API 토큰 비용과 무관하다.** 모델에 실려 가는 것은 매 요청마다
`agent.py`가 최근 N턴만 골라낸 결과이므로, DB에 무엇이 얼마나 쌓여 있든 요청 크기는 변하지 않는다.
**저장 계층과 전송 계층을 분리해서 봐야 한다.**

### 6.2 소켓 — 봇은 서버가 아니라 클라이언트다

```
$ lsof -i -a -p 96826 -n -P
TCP 172.30.1.40:55401 -> 162.159.133.234:443 (ESTABLISHED)
TCP 172.30.1.40:55408 ->  34.149.149.213:443 (ESTABLISHED)
TCP 172.30.1.40:55444 ->  160.79.104.10:443 (ESTABLISHED)
```

각 상대를 DNS로 역추적해 정체를 확정했다.

| 상대 IP | 정체 | 역할 |
|---|---|---|
| 162.159.133.234 | `gateway.discord.gg` | 디스코드 게이트웨이 (WSS 상주 연결) |
| 160.79.104.10 | `api.anthropic.com` | Claude API 호출 |
| 34.149.149.213 | `apac.api.smith.langchain.com` | LangSmith 관측(트레이스 전송) |

**가장 중요한 관찰: `LISTEN` 상태 소켓이 하나도 없다.**

- 유휴(A) 상태엔 디스코드 연결 **1개만** 유지된다. 나머지 둘은 작업할 때 생겼다.
- 셋 다 화살표가 **밖으로 향한다**(`->`). 봇은 스스로 나가서 접속하는 **클라이언트**이고,
  **외부에서 접속해 들어올 문(LISTEN)이 없다.**
- 셋 다 **443 포트**다. 전부 TLS로 암호화된다.

이것이 Phase 3(WireShark)의 출발점이다. 봇에는 캡처할 "서버 트래픽"이 애초에 존재하지 않으므로,
`/health` HTTP 엔드포인트를 추가해 **문을 하나 내야** 다른 컴퓨터의 클라이언트가 접속해 오는
평문 HTTP를 캡처할 수 있다. 위 세 연결은 HTTPS 쪽 분석 대상이 된다 —
암호화되어 내용은 안 보이지만 **핸드셰이크와 접속처(SNI)는 보인다**는 대비를 보여주기에 적합하다.
이번 절에서 IP와 도메인을 확정해 둔 것이 그대로 Phase 3의 캡처 필터가 된다.

## 7. 종합 분석

### 7.1 A / B / C 비교

| 항목 | A (유휴) | B (작업 중) | C (작업 후) | 해석 |
|---|---|---|---|---|
| STAT | S | S | S | 항상 대기 상태 — I/O bound |
| %CPU | 0.0 | 0.1 | 0.0 | 계산이 아니라 기다림이 일의 본체 |
| 스레드 | 4 | 8 | 8 | 작업 시 확장, 이후 반납 안 함(재사용) |
| 자식 프로세스 | 0 | 0* | 0 | *별도 감시로 4쌍 생성·소멸 포착 |
| 소켓 | 1 | 4 | 3 | 유휴엔 디스코드만, 작업 시 Claude·LangSmith 추가 |
| RSS | 127 MB | 138 MB | 142 MB | §5.4 참고 (1차 실행은 반대 추세) |

### 7.2 이 서버의 성격

**I/O bound 프로세스다.** 세 시점 모두 `S` 상태에 CPU 0%대였고, 시간의 대부분을
디스코드·Claude·노션의 응답을 기다리는 데 쓴다. 따라서 CPU를 늘려도 빨라지지 않는다.

**병목은 명확히 하나다: 노션 MCP 호출마다 Node 런타임을 새로 띄우는 비용.**
타임라인에서 도구 호출 1건당 1~2초, 4번 호출에 총 17초가 걸렸다. MCP 세션을 상주시키면
줄일 수 있는 지점이며, 지금은 구조 단순화를 위해 의도적으로 감수하고 있는 비용이다.

**메모리는 문제가 아니다.** 142MB는 상주 봇으로 평범하고, 힙 단편화도 11%로 양호하다.
다만 SQLite가 제곱으로 커지는 구조라 **디스크는 장기적으로 관리가 필요하다.**

### 7.3 관측이 없었으면 몰랐을 것들

- 자식 프로세스는 **일반적인 방법으로는 보이지 않는다.** 세 번의 스냅샷 모두 "자식 0개"였고,
  0.2초 감시 루프를 만든 뒤에야 실체가 드러났다. **관측 도구의 시간 해상도가 관측 대상의
  수명보다 짧아야 한다** — 이 실습에서 얻은 가장 실용적인 교훈이다.
- 이름으로 프로세스를 세면 틀린다. 실습 중 `notion-mcp-server` 프로세스가 6쌍 떠 있었는데
  전부 **다른 프로그램(Claude Code)이 띄운 것**이었다. `pkill`로 지웠다면 무관한 프로그램을
  죽일 뻔했다. 반드시 **PPID로 혈통을 확인**해야 한다.
- `pgrep -f "python.*run.py"`는 봇을 실행한 **쉘까지** 잡는다. 그 쉘의 명령줄 문자열 안에
  `python run.py`가 들어 있기 때문이다. 죽이기 전에 `pgrep`으로 목록부터 확인하는 습관이 필요하다.

## 8. 실습 중 발견한 버그 (부수적 성과)

프로세스 관측을 위해 봇을 돌리던 중, 노션 관련 요청이 전부
`처리 중 문제가 생겼어요, 아가씨. (BadRequestError)` 로 실패했다. 관측 도구로 원인을 추적했다.

**증상 → 원인 추적**

1. 봇 로그에 `UserWarning: Received only system message(s)` — Claude에 **시스템 메시지만** 전송됨
2. `scripts/peek_memory.py`로 저장된 대화를 열어보니, 실패한 턴들이
   최종 답변 없이 `...AI, Tool` 에서 **끊겨 있었다**
3. `agent.py`가 최근 3개 **메시지**만 모델에 싣되 `start_on="human"` 옵션을 쓰고 있었는데,
   도구를 n번 호출하면 그 턴의 메시지는 **2n+1개**로 불어난다.
   도구를 2번만 써도 마지막 3개가 `[Tool, AI, Tool]`이 되어 사람 메시지가 창 밖으로 밀려나고,
   `start_on="human"`이 시작점을 찾지 못해 **전부 버리고 빈 목록**을 반환했다

**검증** — 실제로 봇을 죽인 상태(메시지 14개)를 DB에서 꺼내 양쪽 코드에 먹여 비교했다.

| | 모델에 전달된 메시지 |
|---|---|
| 수정 전 | **0개** → BadRequestError |
| 수정 후 | 10개 (사람 발화 2개 온전히 포함) |

**수정** — 자르는 단위를 "메시지 개수"에서 **"대화 턴"**으로 바꿨다.
마지막 사람 메시지부터 끝까지는 통째로 유지하므로, 도구를 몇 번 호출하든 빈 목록이
구조적으로 발생할 수 없고 `tool_use`↔`tool_result` 짝도 자동 보존된다. (`secretary/agent.py`)

수정 후 실전 확인: 노션 도구를 **3번 연속** 호출하고도 최종 답변까지 정상 완주했다.

> 관측 도구를 만들다가 관측 대상의 버그를 찾은 셈이다.
> 이는 부수적 성과가 아니라 **관측 가능성(observability)이 왜 필요한가**에 대한 실증이다.
> 봇은 예외를 삼키고 `(BadRequestError)`라는 한 줄만 남겼기 때문에, 디스코드 화면만 봐서는
> 영원히 원인을 알 수 없었다. 로그·저장 상태·프로세스를 함께 들여다볼 수 있어야 진단이 가능하다.
